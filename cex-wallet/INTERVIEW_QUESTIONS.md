## 参考答案要点

### 第一部分：项目架构与基础

#### 1.2 模块与端口
| 模块 | 端口 | 功能 |
|------|------|------|
| wallet | 3000 | 用户钱包 API |
| signer | 3001 | 地址生成/签名 |
| evm_scan | 3002 | EVM 链扫描 |
| db_gateway | 3003 | 数据库网关 |
| risk_control | 3004 | 风控评估 |

#### 1.5 启动顺序
```
db_gateway → risk_control → signer → wallet → scan
```
原因：db_gateway 是基础服务，其他模块依赖它访问数据库；risk_control 和 signer 提供签名服务；wallet 依赖前两者；scan 最后启动因为依赖数据库记录。

#### 1.7 数据库隔离原因
- 统一的权限控制和审计
- 防止 SQL 注入
- 统一的签名验证机制
- 便于数据访问日志记录

### 第二部分：Wallet 模块

#### 2.4 提现流程
```
1. 用户请求 → 参数验证
2. 查询代币信息（精度、手续费）
3. 检查用户余额
4. 创建提现请求 → 风控评估
5. 风控通过 → 选择热钱包
6. 构建交易 → 请求风控签名
7. 生成 wallet 签名 → 调用 Signer 签名
8. 发送交易到链
9. 创建 Credit 流水
10. 更新提现状态
```

#### 2.7 策略模式
提现处理器使用**策略模式**：
- `IWithdrawHandler` 接口定义统一方法
- `EvmWithdrawHandler` 和 `SolanaWithdrawHandler` 实现不同链逻辑
- `WithdrawHandlerFactory` 根据链类型返回对应处理器

好处：易于扩展新链，代码复用，职责清晰。

#### 2.8 双签名机制
```
1. Wallet 生成 operation_id 和 timestamp
2. 请求风控评估 → 获取 risk_signature
3. Wallet 使用私钥生成 wallet_signature
4. 调用 Signer 时携带两个签名
5. Signer 验证两个签名通过后才执行交易签名
```

### 第三部分：Signer 模块

#### 3.4 HD 钱包
HD（Hierarchical Deterministic）钱包：从一个种子可以派生出多个子密钥。

**EVM 路径**：`m/44'/60'/0'/0/0`
- 44: BIP44 标准
- 60: ETH 币种代码
- 0/0/0: 账户/变更/索引

**Solana 路径**：`m/44'/501'/0'/0'`
- 501: Solana 币种代码

#### 3.7 双重签名
需要风控签名 + wallet 服务签名，Signer 验证两者通过后才签名交易。
- 防止单点被攻破导致资金损失
- 风控可以拦截可疑交易

#### 3.8 密钥派生
```
助记词 + 密码 → PBKDF2 → 种子 (seed)
种子 → BIP32 HDKey → 派生路径 → 私钥
私钥 → secp256k1 → 公钥 → 地址
```

### 第四部分：Scan 扫描模块

#### 4.8 区块链重组
当区块链发生分叉时，之前确认的区块可能被丢弃，称为重组。

**处理流程**：
1. 检测：比对数据库区块哈希和链上哈希
2. 验证父区块连续性
3. 寻找共同祖先
4. 回滚：删除孤立区块及交易
5. 重新扫描新区块链

#### 4.9 重组回滚流程
```
1. detectReorg() - 哈希比对检测
2. validateParentChain() - 验证父区块连续性
3. findCommonAncestor() - 寻找共同祖先
4. rollbackToCommonAncestor() - 回滚
   - 删除 Credit 记录
   - 标记区块为 orphaned
   - 删除可回滚交易
5. 重新扫描新区块链
```

#### 4.11 性能优化
- 批量扫描（batch 模式）
- 历史分析模式（finalized 区块）
- Bloom 过滤器快速筛选
- 并行处理独立任务

### 第五部分：DB Gateway 模块

#### 5.4 验证流程
```
1. 解析请求体
2. 验证 operation_id 是否已使用
3. 验证时间戳（5 分钟窗口）
4. 验证业务签名（Ed25519）
5. 验证风控签名（敏感操作）
6. 执行数据库操作
```

#### 5.5 Operation ID
UUID 唯一标识每次操作，防止重放攻击。记录在 `used_operation_ids` 表，5 分钟后过期清理。

### 第七部分：安全与加密

#### 7.4 Ed25519 签名
```
签名：signature = sign(message, privateKey)
验证：verify(message, signature, publicKey) → true/false
```

Ed25519 优势：
- 64 字节签名，短小
- 确定性签名（相同消息总是产生相同签名）
- 抗侧信道攻击

#### 7.5 PBKDF2
Password-Based Key Derivation Function 2
系统用于从助记词 + 密码派生种子，增加暴力破解成本。

### 第八部分：区块链知识

#### 8.4 EIP-1559 vs Legacy
| 特性 | Legacy | EIP-1559 |
|------|--------|----------|
| Gas 价格 | gasPrice | baseFee + priorityFee |
| 费用估算 | 困难 | 更可预测 |
| 交易类型 | type: 0 | type: 2 |

#### 8.6 Solana vs EVM
- **EVM**: 账户余额模型，状态存储于合约
- **Solana**: 账户数据模型，程序无状态

#### 8.7 ATA
Associated Token Account：SPL Token 的用户代币账户，由用户地址和代币 Mint 地址派生。

### 第九部分：场景设计

#### 9.4 高并发充值处理
```
1. 多实例扫描器（分区段扫描）
2. 消息队列缓冲（Kafka/RabbitMQ）
3. 批量写入数据库
4. 幂等性保证（operation_id）
5. 分布式锁防止重复处理
```

#### 9.6 冷热钱包分离
```
热钱包（5%）：日常提现，在线签名
温钱包（15%）：自动归集，定时补充热钱包
冷钱包（80%）：离线存储，多签管理

归集策略：
- 用户地址余额 > 阈值 → 自动归集到温钱包
- 温钱包余额 > 阈值 → 手动转入冷钱包
```
